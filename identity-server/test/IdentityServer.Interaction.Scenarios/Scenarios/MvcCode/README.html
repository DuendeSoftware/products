WebClientCodeFlow — MVC Authorization Code Flow + PKCE

Demonstrates an MVC web application authenticating users through Duende IdentityServer using the OAuth 2.0 Authorization Code flow with PKCE, then calling a protected API using the resulting access token.

Services

Three in-process hosts start when this scenario runs:

Service Name Role
Duende IdentityServer identity-server Issues tokens, manages sessions
MVC web application webapp Client — triggers login and calls the API
Protected API api Resource server — requires a valid bearer token

Links to all three services appear in the DevPortal once the scenario is running.

Expected user journey

  1. Navigate to the webapp link in the DevPortal.
  2. The app requires authentication — you are redirected to IdentityServer’s login page.
  3. Sign in with alice / alice (or bob / bob).
  4. IdentityServer issues an authorization code and redirects back to the webapp. PKCE ensures the code cannot be exchanged by a third party.
  5. The webapp exchanges the code for an ID token, access token, and refresh token.
  6. The webapp displays your profile claims (name, email, address, website) fetched from the UserInfo endpoint.
  7. The webapp uses the access token as a bearer token to call the api and displays the response.
  8. You can sign out from the webapp, which triggers a front-channel logout notification back to IdentityServer.

Consent is disabled for this client — you go directly from login to the app.

Scopes requested

Scope Type Purpose
openid Identity Required for OIDC — includes the subject identifier
profile Identity Name, given name, family name, website
email Identity Email address and verification status
custom.profile Identity Name, email, location, and address claim
resource1.scope1 API Access to Resource 1
resource2.scope1 API Access to Resource 2 (also includes name and email claims in the access token)
offline_access Special Refresh token — allows token renewal without re-authentication

Key implementation details

  • Response type: code (Authorization Code flow — no tokens in the browser URL)
  • PKCE: enabled — a code verifier/challenge pair is generated per request
  • Client secret: secret (shared secret between the webapp and IdentityServer)
  • Token storage: tokens are saved to the authentication cookie so the webapp can use them for API calls
  • UserInfo endpoint: claims are fetched from the UserInfo endpoint after token exchange
